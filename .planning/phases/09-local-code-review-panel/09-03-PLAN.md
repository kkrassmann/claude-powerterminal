  <action>
    **1. Add per-hunk accept/reject buttons to DiffViewerComponent:**

    After diff2html renders HTML into the container, use DOM manipulation to inject control buttons:
    - Find all hunk headers (`.d2h-info` elements in the rendered HTML) — each corresponds to a DiffBlock
    - Before each hunk header, inject a div with: "Accept" button (green), "Reject" button (red)
    - Track hunk states in the component: `hunkStates: ReviewHunkState[]` (pending/accepted/rejected) per file
    - **Accept action:** Mark hunk as accepted (visual only — green check overlay on the hunk, dim the lines slightly). No git operation needed per user decision.
    - **Reject action:**
      1. Store the current hunk's forward patch content (for undo)
      2. Call `codeReviewService.rejectHunk(cwd, patchContent)` where patchContent is the reversed patch reconstructed from the DiffBlock (see RESEARCH.md Pattern 4: buildHunkPatch function)
      3. On success: show timed "Undo" toast/button (10 seconds) at the bottom of the diff viewer
      4. Re-fetch the full diff after reject (stale diff pitfall — RESEARCH.md Pitfall 2) and re-render
      5. The rejected hunk disappears from the new diff naturally
    - **Undo action:** Apply the stored forward patch via `codeReviewService.rejectHunk(cwd, forwardPatch)` — but this time use a regular `git apply` (not reverse). Add a method to CodeReviewService: `applyPatch(cwd, patchContent): Promise<{success, error?}>`. Then re-fetch diff.
    - **File-level bulk operations:** Add "Accept All" and "Reject All" buttons in the file header area of the diff viewer
      - "Accept All": mark all hunks as accepted
      - "Reject All": call `codeReviewService.rejectFile(cwd, filename)`, then re-fetch diff
    - "Reviewed" toggle: Add a checkbox "Reviewed" in the file header. When checked, emit event to parent to mark file as reviewed (collapses back in file tree per user decision).

    Add output events to DiffViewerComponent:
    - `hunkRejected: EventEmitter<void>` — parent re-fetches diff
    - `fileRejected: EventEmitter<void>` — parent re-fetches diff
    - `fileReviewed: EventEmitter<{ filename: string, reviewed: boolean }>` — parent updates file tree

    **Undo toast styling:** Fixed position at bottom-center of the diff viewer, `background: #313244`, `color: #cdd6f4`, border-left with `#f38ba8` (red). "Undo" button in white. Auto-dismiss after 10 seconds using setTimeout. Clear timeout if undo is clicked.

    **2. Create CommentSidebarComponent:**

    Component structure:
    - Inputs: `sessionId: string`, `filename: string`, `comments: ReviewComment[]`
    - Outputs: `sendNow: EventEmitter<ReviewComment>`, `sendSummary: EventEmitter<ReviewComment[]>`, `commentAdded: EventEmitter<{line: number, text: string}>`, `commentResolved: EventEmitter<string>` (commentId)
    - Display: list of comments for current file, each showing:
      - Line number (clickable — emits event to scroll diff to that line)
      - Comment text
      - Checkbox for resolved (strikethrough when resolved)
      - "Send Now" button (paper plane icon) — emits `sendNow` with the comment
      - Delete button (x)
    - Bottom section: "Send Summary" button that sends all unresolved comments for the file

    CSS: Width 300px, right side of the diff area, `background: #181825` (mantle), `border-left: 1px solid #313244`. Comment cards: `background: #1e1e2e`, `border-left: 3px solid #89b4fa` (blue). Resolved: dimmed, strikethrough. Scrollable with `overflow-y: auto`.

    **3. Add comment input to diff viewer:**

    When user clicks a line number in the diff (`.d2h-code-linenumber`), show a small inline input form:
    - Text input + "Add" button
    - On submit: emit event to code-review-panel which calls `codeReviewService.addComment()`
    - Close on Escape or clicking elsewhere
    - Input positioned below the clicked line

    **4. Update CodeReviewPanelComponent to orchestrate everything:**

    - Add the comment sidebar to the layout (right of diff viewer, conditionally visible when comments exist for current file)
    - Handle `lineClicked` from diff-viewer → show comment input
    - Handle comment add → call service, refresh comment list
    - Handle `sendNow` → format prompt "Review-Feedback fuer [filename]:\n- Zeile [line]: [comment]\n" and write to terminal PTY via `ptyManagerService.writeToSession(sessionId, prompt)` or IPC/HTTP equivalent
    - Handle `sendSummary` → collect all unresolved comments for file, format as structured prompt, write to terminal
    - Handle hunk/file rejected → re-fetch diff, re-render
    - Handle file reviewed → update file tree (mark as reviewed, collapse)
    - Re-fetch diff after any reject operation (critical for avoiding stale diff line numbers)

    **IMPORTANT for terminal prompt injection:**
    - In Electron: use `window.electronAPI.invoke(IPC_CHANNELS.PTY_WRITE, sessionId, prompt)`
    - In remote browser: use WebSocket or HTTP. Check existing PtyManagerService for the write pattern. The WebSocket protocol already supports writing to a session — the panel should use the same mechanism.
    - Format: `"Review-Feedback fuer [filename]:\n- Zeile [line]: [comment]\n"` per user decision
