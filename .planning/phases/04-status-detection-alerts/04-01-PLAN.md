Create `electron/status/ansi-strip.ts`:
- Export `stripAnsi(text: string): string` function
- Use inline ANSI regex from ansi-regex project (MIT): `/[\u001b\u009b][[()#;?]*(?:[0-9]{1,4}(?:;[0-9]{0,4})*)?[0-9A-ORZcf-nq-uy=><~]/g`
- Do NOT use the strip-ansi npm package (ESM-only since v7, this project is CommonJS)

Create `electron/status/status-detector.ts`:
- Export type `TerminalStatus = 'WORKING' | 'THINKING' | 'WAITING' | 'ERROR' | 'DONE'`
- Export type `StatusChangeCallback = (sessionId: string, status: TerminalStatus, previousStatus: TerminalStatus) => void`
- Export class `StatusDetector` with constructor `(sessionId: string, onStatusChange: StatusChangeCallback)`
- Internal state: `status: TerminalStatus` (initial 'WORKING'), `lastOutputTime: number`, `idleTimer: NodeJS.Timeout | null`, `recentOutput: string` (sliding window, max 500 chars)
- **WAITING_PATTERNS** (per CONTEXT.md, hardcoded): `❯` at end of output, `? ... (Y/n)` questions, `? ... (y/N)`, `Do you want to proceed` (case-insensitive), `[Y/n]`
- **WORKING_PATTERNS**: `● ` tool call indicator
- **ERROR_PATTERNS**: `Error:`, `Interrupted`, `ENOENT`, `EACCES`
- `processOutput(data: string)`: Strip ANSI via `stripAnsi`, append to sliding window (trim to 500 chars from end), reset idle timer timestamp, check patterns in priority order: ERROR > WAITING > WORKING (any new output = WORKING if no other pattern matches). Clear recent output buffer when transitioning to WORKING from WAITING/ERROR (prevents stale pattern matches).
- `processExit()`: Clear idle timer, transition to DONE
- `destroy()`: Clear idle timer
- `getStatus(): TerminalStatus`: Returns current status
- Idle timer: `setInterval` every 1000ms, if status is WORKING and `Date.now() - lastOutputTime >= 5000`, transition to THINKING (per CONTEXT.md: 5-second threshold)
- `transition(newStatus)`: Only call callback if status actually changed (prevent flooding)

Update `src/shared/ws-protocol.ts`:

- Extend `ServerMessage` union with `| { type: 'status'; status: TerminalStatus }` — add as last union member, preserving all existing types

Modify `electron/websocket/ws-server.ts`:
- Import `TerminalStatus` from `../../src/shared/ws-protocol`
- Add export function `broadcastStatus(sessionId: string, status: TerminalStatus): void` that:
  - Checks `wss` is not null
  - Iterates `wss.clients`, finds clients where `(ws as any).sessionId === sessionId` and `ws.readyState === WebSocket.OPEN`
  - Sends `JSON.stringify({ type: 'status', status })` to matching clients
- Also on WebSocket connection (after scrollback replay), send current status for the session if a StatusDetector exists. Add export function `getStatusDetectors(): Map<string, StatusDetector>` or accept a getter callback. Simpler approach: add a `statusDetectors` Map alongside `scrollbackBuffers`, with getter `getStatusDetectors()`. On connection, after buffer replay, look up detector and if it exists, send `{ type: 'status', status: detector.getStatus() }` so new clients immediately know the current state.

Modify `electron/ipc/pty-handlers.ts`:
- Import `StatusDetector` from `../status/status-detector`
- Import `broadcastStatus` from `../websocket/ws-server`
- Create a `statusDetectors` Map alongside `ptyProcesses`: `const statusDetectors = new Map<string, StatusDetector>()`
- Export `getStatusDetectors(): Map<string, StatusDetector>` for ws-server to use on new connections
- In PTY_SPAWN handler, after creating scrollback buffer:
  - Create `new StatusDetector(sessionId, (sid, status, prev) => { broadcastStatus(sid, status); })`
  - Store in `statusDetectors` map
- In `ptyProcess.onData` handler (both in PTY_SPAWN and PTY_RESTART):
  - After appending to scrollback buffer, call `statusDetectors.get(sessionId)?.processOutput(data)`
- In `ptyProcess.onExit` handler (both in PTY_SPAWN and PTY_RESTART):
  - Call `statusDetectors.get(sessionId)?.processExit()` before deleting from map
  - Call `statusDetectors.get(sessionId)?.destroy()` then `statusDetectors.delete(sessionId)` in cleanup
- In PTY_KILL handler: Destroy and remove StatusDetector from map
- In PTY_RESTART handler:
  - Destroy old StatusDetector before killing old PTY
  - Create new StatusDetector after spawning new PTY
  - Wire new PTY onData to new StatusDetector

In `ws-server.ts` connection handler:

- After scrollback buffer replay completes (or if no buffer), send initial status: `const detector = getStatusDetectors().get(sessionId); if (detector) { safeSend({ type: 'status', status: detector.getStatus() }); }`
